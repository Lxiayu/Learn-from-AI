## 2. 核心结论

### 2.1 现在是否还可以只做前端

结论：`不能只做前端了`

更准确地说：

- 只靠前端，仍然可以继续验证视觉、路由、交互和局部状态。
- 但如果要真正测试“引导式学习”这件事是否成立，仅靠前端已经不够。

原因很直接：

`guided-learning-coach` 不是一个普通内容应用，它的核心价值不在页面，而在“学习工作流引擎”。

而这个引擎至少需要以下三类能力：

1. `可持续状态`
   - 学习路线
   - 当前知识点
   - 当前会话
   - 掌握度
   - 复习计划

2. `可计算逻辑`
   - 学习路线生成
   - 问题阶梯推进
   - 纠错与补讲
   - 掌握度判定
   - 复习任务生成

3. `可恢复数据`
   - 中断恢复
   - 本地优先保存
   - 未来同步

这三类能力只做前端 mock，最多只能证明“看起来像”，不能证明“真的能学”。

### 2.2 现在是不是就要做完整后端

结论：`也不需要一上来就做完整后端体系`

最合理的下一步不是“只做前端”，也不是“马上做完整生产级后端”，而是：

`前端 + 本地数据库 + 本地 FastAPI + 最小后端能力` 同步推进

建议的工程顺序：

1. 前端继续优化和逻辑确认
2. 先落本地数据库和状态仓储
3. 再接本地 FastAPI 与最小后端能力
4. 最后再扩展云同步、完整用户体系和长期数据

补充说明：

- 当前阶段默认以 **本地优先后端** 为实施方式
- CloudBase 仍保留为未来迁移目标，但不再是当前开发阻塞项
- 相关架构决策见 [2026-04-05-local-first-backend-and-cloud-migration-design.md](/Users/xia/program/Learn/docs/superpowers/specs/2026-04-05-local-first-backend-and-cloud-migration-design.md)

## 3. 当前已具备的能力

### 3.1 已具备的产品外壳

当前前端已经具备：

- 5 个一级页面结构
- 关键二级页面结构
- 中英文切换
- Stitch 风格视觉基础
- 基本路由与无奇怪转场

这些能力足以证明：

- 信息架构方向是对的
- 页面组织方向是对的
- UI 容器层已经基本可用

### 3.2 当前仍然只是“外壳”的部分

以下内容目前仍然主要是视觉和展示层：

- 学习路线
- 学习对话
- 复习计划
- 测验
- 掌握度
- 数据统计
- 成就

也就是说：

- 页面已经存在
- 但真正驱动这些页面变化的状态机和数据结构还没有落地

## 4. 对照原始 skill 的功能差距

### 4.1 学习启动向导缺失

原始 skill 明确要求：

- 收集学习主题
- 收集目标结果
- 可选收集当前水平
- 可选收集可投入时间
- 可选收集是否需要评价

当前应用缺口：

- 没有完整的学习启动入口
- 没有“第一次学习时如何创建学习目标”的产品流程
- 没有把用户输入转成路线生成请求的表单流

优先级：`P0`

### 4.2 路线确认机制缺失

原始 skill 强调：

- 先生成学习路线图
- 用户确认后再进入正式学习

当前应用缺口：

- 有 `Roadmap` 页，但没有“确认路线”动作
- 没有“未确认路线”与“已开始路线”的状态区分

优先级：`P0`

### 4.3 当前任务卡缺失

原始 skill 的输出模板里，`当前任务卡` 是明确存在的。

它应该至少包含：

- 本轮主题
- 本轮先理解什么
- AI 重点追问什么
- 本轮通过标准
- 卡住时 AI 如何帮助

当前应用缺口：

- 没有一个真正可执行的任务卡层
- Home 与 Chat 的关系还没有被任务系统串起来

优先级：`P0`

### 4.4 单知识点学习状态机缺失

原始 skill 明确规定了单知识点循环：

1. 认知唤起
2. 系统讲解
3. 理解验证
4. 纠错与补讲
5. 收束

当前应用缺口：

- `Chat` 页还不是状态机驱动
- 问题并没有真正按 `解释 -> 举例 -> 比较 -> 迁移` 推进
- 也没有“当前正在第几步”的明确数据结构

优先级：`P0`

### 4.5 纠错与补讲引擎缺失

原始 skill 的核心差异化之一是：

- 先鼓励，再拆解
- 识别误解点
- 换一种方式重讲
- 必要时切到提示 + 重讲

当前应用缺口：

- 有页面，但没有真正的错误分类与补讲策略
- 无法支撑“不是重复回答，而是精准纠错”

优先级：`P0`

### 4.6 掌握度模型缺失

原始 skill 把掌握度拆成：

- 识别
- 解释
- 举例
- 迁移

当前应用缺口：

- 没有真正的 mastery model
- 也没有“达到基础掌握即可前进”的状态规则
- 测验、复习、路线推进之间还没有共用的掌握度来源

优先级：`P0`

### 4.7 本轮小结缺失

原始 skill 要求每轮结束输出：

- 已掌握
- 仍模糊
- 下一步建议

当前应用缺口：

- 学习会话结束后没有真正的 session close
- 不能自然衔接到复习计划

优先级：`P1`

### 4.8 动态复习生成缺失

原始 skill 不是简单做固定的 `1/3/7/14` 天提醒，而是强调：

- 根据掌握深度调整
- 根据失败模式调整
- 优先复习相邻概念混淆、迁移失败、开放题冻结等情况

当前应用缺口：

- `Review` 页存在，但更像复习容器
- 复习任务还不是动态生成出来的

优先级：`P0`

### 4.9 发散学习分支缺失

你最初的想法里明确提到：

- 用户对某个知识点感兴趣时，支持发散探索
- 但不能打断主线

原始 skill 也明确规定：

- 默认先记录为 pending branch
- 主线结束后再展开

当前应用缺口：

- 没有 branch 记录
- 没有“主线 / 支线”切换机制

优先级：`P1`

### 4.10 中断恢复缺失

原始 skill 与 PRD 都强调学习状态管理与中断恢复。

当前应用缺口：

- 没有真实会话存储
- 没有恢复上一次学习位置
- 退出 Chat 后无法自然回到原来的知识点上下文

优先级：`P0`

### 4.11 本地学习数据库缺失

你原始的 app 化设想里明确提到：

- 学习数据库
- 存储有用内容
- 方便持续学习

当前应用缺口：

- 还没有真正落地本地数据库
- 所有学习成果无法长期沉淀

优先级：`P0`

### 4.12 推荐阅读系统缺失

你最初也提到：

- 推荐人文社科、传记、经典书籍
- 帮助扩展思维，补全自我

当前应用缺口：

- 完全没有相关阅读层

但这里不建议当前优先做。

优先级：`P2`

## 5. 哪些功能只靠前端不够

### 5.1 必须接本地数据库的能力

以下能力如果不接本地数据库，几乎无法被真正验证：

- 学习路线持久化
- 当前会话持久化
- 中断恢复
- 复习计划持久化
- 成就进度
- 学习统计
- 设置与同步偏好

建议对应：

- `SQFlite`
  - 学习路线
  - 阶段
  - 知识点
  - 复习计划
  - 复习任务
  - 统计
  - 成就
- `Hive`
  - Chat Session
  - Chat Message
  - Learning Log
- `SharedPreferences`
  - 语言
  - 提醒
  - 同步设置
  - 界面偏好

### 5.2 必须接后端或 AI 服务的能力

以下能力如果完全不接后端，最多只能做 demo：

- 路线生成
- 苏格拉底式问题生成
- 智能纠错
- 补讲内容生成
- 掌握度诊断
- 动态复习任务生成
- 推荐阅读生成

也就是说，这一批功能不能永远停留在前端静态文案里。

### 5.3 可以先用本地引擎过渡的能力

以下能力可以先不接真实后端，而通过本地状态机 + 假引擎做一版：

- 当前任务卡生成
- 会话步骤推进
- 本轮小结生成
- 主线 / 支线分流
- 基础掌握判断
- 测验状态流转

这类能力适合先做成：

`前端状态机 + 本地规则引擎`

然后后续再替换成：

`后端服务 + AI 推理`

## 6. 当前建议的技术推进策略

### 6.1 推荐路线

推荐路线不是：

- 继续只做前端
- 或者直接扑向完整后端

而是：

`本地优先 MVP`

分三层推进：

### Layer 1：前端状态层

目标：

- 页面不再只是展示 mock 文案
- 所有关键页面接入真实 provider 和本地状态流

需要落的核心对象：

- learning session
- roadmap progress
- review tasks
- mastery signals
- settings

### Layer 2：本地数据库层

目标：

- 用户退出应用后还能恢复
- 真正能测“学习流程是否连续”

必须先落的数据：

- 当前路线
- 当前会话
- 当前知识点
- 复习计划
- 学习日志
- 设置

### Layer 3：最小后端层

目标：

- 把产品最核心的“AI 生成与判断”能力接入真实服务

建议第一批后端只做 4 个能力：

1. 路线生成
2. 问题生成
3. 掌握度评估
4. 复习计划生成

先不要一开始就做：

- 完整账号体系
- 完整云同步
- 完整运营后台
- 推荐阅读引擎

## 7. 推荐的功能优先级

### P0：必须尽快开始

这些功能决定产品是否真的成立：

1. 学习启动向导
2. 路线确认机制
3. 当前任务卡
4. Chat 学习状态机
5. 纠错与补讲引擎
6. 掌握度模型
7. 动态复习生成
8. 中断恢复
9. 本地学习数据库

### P1：紧接着做

这些功能决定产品是否顺滑、是否有明显成长感：

1. 本轮小结
2. 发散学习分支
3. 成就状态联动
4. 学习数据看板与真实数据联动
5. 独立 `Review Schedule`
6. 独立 `Settings Center`

### P2：后续增强

这些功能重要，但不应抢在核心闭环前面：

1. 推荐阅读系统
2. 语音输入
3. 图片输入的真实 OCR/识别链路
4. 会员体系
5. 云同步与冲突解决
6. 第三方登录

## 8. 当前最建议先做的 3 件事

### 8.1 先做本地数据骨架

理由：

- 不做本地存储，所有会话都只是临时展示
- 无法真正验证学习闭环

### 8.2 再做学习会话状态机

理由：

- 这是 skill 真正的核心
- 没有它，Chat 永远只是“对话页面”，不是“引导式学习引擎”

### 8.3 再接最小 AI 后端

理由：

- 路线生成、问题生成、掌握度评估都需要真正的服务支撑
- 但不需要一开始就做完整后端平台

---

## 9. 最终判断

最终结论如下：

1. 现在确实不适合继续只做前端
2. 现在也不应该直接做完整生产级后端
3. 最合适的阶段切换是：
   - `前端优化继续`
   - `本地数据库开始落地`
   - `最小后端能力同步启动`
4. 当前真正缺的不是更多页面，而是让学习流程“会运行、会记住、会判断、会复习”

这也是从“漂亮原型”进入“真正可用应用”的关键分界线。
